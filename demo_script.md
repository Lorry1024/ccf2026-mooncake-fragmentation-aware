# 5分钟作品展示视频录制方案

## 一、录制目标

视频需要让评审在5分钟内确认四件事：项目解决了什么真实问题，代码如何实现，效果如何验证，以及成果是否已经形成公开、可复现、可评审的提交闭环。不要把视频录成逐页朗读PPT，也不要花大量时间展示文件目录。每一个画面都应当对应一个明确结论。

建议采用“PPT讲解+GitHub证据+终端复现”的组合方式。PPT负责解释问题和方案，GitHub负责证明PR与CI，终端负责证明复现实验确实能够运行。摄像头不是必须项，稳定清晰的旁白和可读画面更重要。

## 二、推荐录制参数

- 软件：OBS Studio，使用“显示器采集”或“窗口采集”。
- 分辨率：1920×1080。
- 帧率：30fps。
- 编码：H.264，视频码率建议8-12Mbps。
- 音频：48kHz，开启基础降噪，录制前先试听10秒。
- 格式：优先录制为MKV，录制完成后由OBS重新封装为MP4，避免异常退出导致文件损坏。
- 鼠标：保留鼠标指针，但移动要慢，避免来回晃动。
- 浏览器：使用浅色主题，缩放80%-100%，关闭书签栏和无关扩展提示。
- 终端：字号至少20pt，预先清空无关输出。

## 三、录制前准备

按以下顺序预先打开窗口或标签页，录制时只做切换，不临时输入网址：

1. 更新版PPT，停在封面。
2. 上游PR：`https://github.com/kvcache-ai/Mooncake/pull/2797`。
3. PR的`Checks`或页面底部“All checks have passed”区域。
4. PR的`Files changed`页面，提前定位`allocation_strategy.h`和测试文件。
5. 比赛材料仓库：`https://github.com/Lorry1024/ccf2026-mooncake-fragmentation-aware`。
6. 终端，工作目录切换到比赛材料仓库根目录。
7. 技术报告PDF，提前定位“验证与效果”章节。

录制前关闭邮件、聊天软件、下载提示和系统通知，检查GitHub页面上是否显示私人邮箱、访问令牌、内网地址或其他不应公开的信息。所有命令至少预演一遍；若现场编译耗时或环境不稳定，可以展示已生成日志，但应明确说明日志来自仓库中的可复现程序。

## 四、5分钟时间轴与讲稿

### 0:00-0:25 项目定位

画面：PPT第1页。

讲稿：

大家好，我们的作品是“Mooncake Store碎片感知优化”，对应CCF 2026开源创新大赛Mooncake赛题2。我们没有修改传输协议，而是优化Mooncake Store Master侧的segment分配决策，让混合尺寸KVCache对象优先尝试真正具有足够连续空间的位置。

### 0:25-0:55 问题背景

画面：PPT第2页。

讲稿：

Mooncake Store长期运行后，不同大小的KVCache对象不断分配和释放，会形成不连续空闲区。原有`free_ratio_first`主要按总空闲比例排序，但总空闲很多不代表存在一个足够大的连续区域。因此，大对象可能先落到不可容纳的segment，触发allocator失败和fallback。

### 0:55-1:35 策略原理

画面：PPT第3页。

讲稿：

我们新增`fragmentation_aware`策略。它沿用Mooncake现有的bounded sampling，不做全局扫描。对每个候选读取总空闲空间和最大连续空闲区域，先用`can_fit`判断当前请求能否直接容纳，再用连续性比例与总空闲比例计算score。可容纳候选优先，score只在相同可容纳状态内排序。preferred、excluded、多副本去重和原有fallback语义都保持不变，最终分配仍由allocator确认。

### 1:35-2:10 工程实现

画面：先展示PPT第4页，再切换PR的`Files changed`。

讲稿：

这不是离线算法原型。PR在9个文件中完成正式接入，包括策略枚举、配置解析、工厂创建、核心候选排序、命令行帮助、单元测试、benchmark和官方文档。这里可以看到核心实现位于`allocation_strategy.h`，测试覆盖碎片反例和preferred语义；启动时通过`--allocation_strategy=fragmentation_aware`显式启用，默认行为不变。

### 2:10-2:40 上游PR与CI证据

画面：PR顶部和“All checks have passed”区域，随后展示PPT第6页。

讲稿：

我们已经向Mooncake上游创建draft PR，编号2797。开发过程中修复了格式、GitHub runner磁盘空间和碎片测试构造问题。最终GitHub Actions结果为26项检查成功、1项按条件跳过，Linux构建、格式检查、文档构建和相关测试均通过。这说明代码已经进入上游项目的真实工程验证流程。

### 2:40-3:30 现场复现实验

画面：终端。

建议预先准备并执行：

```bash
g++ -std=c++17 -O2 repro/topic_aligned_store_scalability_sim.cpp -o topic_aligned_store_scalability_sim
./topic_aligned_store_scalability_sim
```

如果希望同时展示详细指标，可继续执行：

```bash
g++ -std=c++17 -O2 repro/fragmentation_aware_metrics.cpp -o fragmentation_aware_metrics
./fragmentation_aware_metrics
```

讲稿：

现在运行仓库中的确定性复现实验。两个策略面对相同的5个候选segment和6个大对象请求。基线的首选可直接容纳次数是0/6，新策略是6/6；fallback尝试次数从11降到0；平均候选数量都为5.00，说明收益来自排序信号变化，而不是扩大搜索范围。

### 3:30-4:05 结果与开销

画面：PPT第5页或技术报告“专题仿真指标”。

讲稿：

两个策略最终都能达到6/6可容纳，因此新策略没有增加物理容量，而是更快找到已经存在的正确位置。单次决策耗时从121.00ns增加到138.93ns，新增约17.93ns。我们的结论是：策略以可控的纳秒级元数据计算，减少碎片化场景中的错误首选和fallback。这里的数据是策略层确定性验证，不外推为生产集群QPS。

### 4:05-4:35 赛题匹配与提交材料

画面：PPT第7页，随后切换GitHub比赛材料仓库。

讲稿：

成果对应赛题2的Store吞吐性能、可扩展性和HiCache后端适配方向。代码位于真实Store路径，候选规模保持有界，兼容已有副本和fallback语义。公开仓库提供技术报告、PPT、patch、复现源码、日志和最终压缩包，Mooncake fork与上游PR用于复核完整开发历史。

### 4:35-5:00 结论

画面：PPT第8页。

讲稿：

总结来说，我们针对“总空闲比例无法反映连续可用空间”这一具体失效模式，为Mooncake Store实现了可选的碎片感知分配策略。它不改变默认行为，不扩大搜索范围，并已经通过上游CI和确定性复现验证。后续我们将继续补充真实Store压力测试、RDMA环境和SGLang HiCache端到端评测。谢谢各位评审。

## 五、录制后的检查

1. 视频总时长控制在4分40秒到4分58秒，避免超过5分钟。
2. 从头到尾检查一次，确认没有黑屏、窗口切换卡顿或音画不同步。
3. 在手机或普通笔记本上试听，确保人声清楚、终端文字可读。
4. 剪掉编译等待、页面加载和误操作，不添加影响阅读的花哨转场。
5. 导出文件建议命名为`CCF2026_Mooncake_作品展示.mp4`。
6. 将视频上传到GitHub Release、网盘或平台允许的位置，并确认链接无需登录即可访问。

## 六、现场失败的兜底方案

如果终端编译失败或耗时过长，不要在视频中排查环境。直接打开`logs/`中的参考日志，说明“这里展示的是仓库随附的确定性复现日志，源码和编译命令已经公开”，随后继续解释0/6、6/6、11到0和候选数量5.00。视频的目标是证明成果可复现，不是现场展示环境安装过程。
